Ho un aeroporto in cui ho un sistema di riconoscimento facciale IoT basato su edge computing, in particolare ho degli scanner che catturano immagini di 5 tipi:
	• Immagini sfocate
	• Persone ok
	• Persone sospette
	• Persone pericolose
	• Persone sconosciute
Queste immagini vengono inviate a dei nodi edge che le categorizzano sulla base di un loro database locale, nel caso di persone sconosciute l'immagine viene inviata
al cloud (ogni nodo edge ha una VM in cloud dedicata) per un ulteriore processamento con risorse e metodi non accessibili dai nodi edge, 
il risultato della computazione nel nodo cloud ritorna al nodo edge che aggiorna il suo database locale.
Tutte le altre immagini "escono" dal sistema.
Chiamiamo le immagini di persone sconosciute scan di classe C e le altre scan di classe E.

DATI
	• N=30 nodi
	• Il tasso di arrivo è di 42 scan/s ed ogni nodo gestisce 1.4 scan/s
	• La probabilitá che una scansione sia di un utente sconosciuto è PC 
	• La probabilitá di uscire dal sistema è 1-PC
	• Il tempo che un nodo edge impiega per processare una scan di classe E è 500ms 
	• Il tempo che un nodo edge impiega per salvare su db la risposta del cloud, cioè processare una scan di classe C è 100ms
	• Il tempo che il cloud impiega a processare una scan di classe C è 800ms
	• Il QOS è che il tempo di risposta per i job di classe E al nodo edge non deve superare i 3s
	• Dato che abbiamo due tipi di job utilizziamo processor sharing come scheduling sui nodi edge e FIFO sul cloud


IDEE
	• Valutare come cambia il bottleneck se alcuni nodi ricevono piú traffico di altri
	• Considerare la possibilitá di avere autoscaling delle vm cloud al variare del tipo e della quantitá di traffico


VARIABILI DI STATO VERSIONE 1
	• Numero di scan di classe C nel nodo cloud i-esimo
	• Numero di scan di classe E nel nodo edge i-esimo
	• Classe del job i-esimo

VARIABILI DI STATO VERSIONE 2 con autoscaling

	• Numero di scan di classe E nel nodo edge i-esimo

EVENTI VERSIONE 1
    EVENTI AL NODO EDGE
    • Arrivo di una scan nel nodo edge i-esimo (assunzione, tutti i job entrano come classe E)
    • Partenza di una scan di classe dal nodo edge i-esimo

    EVENTI AL NODO CLOUD
    • Arrivo di una scan nodo cloud i-esimo
    • Partenza di una scan nodo cloud i-esimo

EVENTI VERSIONE 2
    EVENTI AL NODO EDGE
    • Arrivo di una scan nel nodo edge i-esimo (assunzione, tutti i job entrano come classe E)
    • Partenza di una scan di classe dal nodo edge i-esimo

    EVENTI AL NODO CLOUD
    • Arrivo di una scan nodo cloud i-esimo
    • Partenza di una scan nodo cloud i-esimo
    • Spawn di una nuova VM nel nodo cloud i-esimo
    • Kill di una VM nel nodo cloud i-esimo


COSA SUCCEDE PER I VARI EVENTI VERSIONE 1
    EVENTI AL NODO EDGE
    • Arrivo di una scan di tipo E nel nodo edge i-esimo
        ○ Aumento del numero di scan di classe E nel nodo edge i-esimo

    • Partenza di una scan di classe E dal nodo edge i-esimo
        ○ Diminuzione del numero di scan di classe E nel nodo edge i-esimo
        ○ Conversione della scan di classe E in scan di classe C con probabilitá PC, schedulare l'evento di arrivo della scan di classe C al nodo cloud i-esimo
        ○ Se la scan rimane di classe E, con probabilità 1-PC, la scan esce dal sistema

    • Arrivo di una scan di classe C nel nodo edge i-esimo
        ○ Aumento del numero di scan di classe C nel nodo cloud i-esimo

    • Partenza di una scan di classe C dal nodo edge i-esimo
        ○ Diminuzione del numero di scan di classe C nel nodo cloud i-esimo
        ○ Convertire la scan di classe C in scan di classe E, la scan esce dal sistema

    EVENTI AL NODO CLOUD
    • Arrivo di una scan nodo cloud i-esimo
        ○ Aumento del numero di scan di classe C nel nodo cloud i-esimo
    • Partenza di una scan nodo cloud i-esimo
        ○ Diminuzione del numero di scan di classe C nel nodo cloud i-esimo, schedulare l'evento di arrivo della scan di classe C al nodo edge i-esimoù
